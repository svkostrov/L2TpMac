## Итерация проверки и исправлений 17.07.2026 (v1.53)

**BR-36 (исправлен). В окне отображались только последние 60 строк PPP/L2TP-лога**
- Причина: GUI в `refresh()` и после очистки лога использовал `tail(logPath, lines: 60)`.
- Исправление: заменено на полное чтение файла лога текущей сессии (`readLog`), лог по-прежнему очищается при запуске приложения и хранится/показывается от запуска до завершения.
- Проверка: приложение запущено с чистым логом (`0` строк), затем выполнены 3 GUI-цикла connect/disconnect. Количество строк в файле росло без обрезки: `44 → 91 → 128`. Accessibility-текст лога в окне содержит те же `128` строк; первая и последняя строки совпадают с `/tmp/l2tp-office-app.log`, нормализованное сравнение полного текста `file.rstrip("\\n") == ui.rstrip("\\n")` — `True`.
- Сеть после проверки: каждое подключение поднимало `ppp0`, ping до `172.16.99.10` проходил; каждое отключение снимало `ppp0`; после каждого отключения GitHub отвечал `HTTP/2 200`.

**BR-37 (исправлен). Menu bar-панель: синяя кнопка «Настройки» и две отдельные кнопки подключения**
- Исправление: «Настройки» перенесены под action-кнопку и сделаны обычной bordered-кнопкой без синего prominent-стиля.
- Исправление: «Подключить» и «Отключить» в menu bar объединены в одну контекстную кнопку: при отключенном состоянии подключает, при подключенном отключает, во время подключения показывает stop-действие.
- Исправление: новая action-кнопка перенесена в одну строку со статусом подключения справа от текста «Отключено/Подключено».

## Новые баги

### Критичные / High

**BR-01. Enter в любом текстовом поле запускает «Подключить»**
- Северити: High. Статус: подтверждён живым тестом.
- Шаги: фокус в поле «Сети» (или любом другом), ввести текст, нажать Enter.
- Ожидание: завершение редактирования поля.
- Факт: стартует подключение (`keyboardShortcut(.defaultAction)` перехватывает Return глобально), появляется админ-диалог. В сочетании с BR-02 запустилось подключение с мусорными настройками.

**BR-02. Нет валидации поля «Сети» (CIDR) → ложный «Подключено»**
- Северити: High. Статус: подтверждён.
- Шаги: режим «Только выбранные сети», ввести `abc-invalid 999.999.999.0/99`, подключиться.
- Ожидание: ошибка валидации до подключения либо ошибка добавления маршрутов.
- Факт: туннель поднялся, статус «Подключено», но `route add` молча упал (`>/dev/null`): в таблице маршрутов НЕТ ни одного маршрута в туннель (проверено `netstat -rn`) — VPN фактически не используется, пользователь об этом не узнает.

**BR-03. Вертикальная обрезка контента окна**
- Северити: High (UI). Статус: подтверждён скриншотами.
- Шаги: режим «Только выбранные сети» (контент выше), окно в размере по умолчанию/уменьшенное.
- Факт: строка статуса («Подключено/Отключено…») частично уходит под заголовок окна (верх букв срезан), нижняя рамка блока «Лог pppd» обрезается краем окна. Причина: заявленный `minHeight: 560` меньше фактической высоты контента; `.windowResizability(.contentSize)` не пересчитывает минимум при динамическом появлении строки «Сети». После ручного увеличения окна обрезка исчезает, при уменьшении возвращается.

### Средние / Medium

**BR-04. Отмена админ-диалога не даёт внятной ошибки**
- Северити: Medium. Статус: disconnect-ветка подтверждена живым тестом; connect-ветка — по коду.
- Факт (disconnect): нажали «Отключить» → «Отмена» в диалоге → статус молча возвращается к «Подключено», никакого сообщения (completion безусловно очищает lastError).
- Факт (connect, по коду): osascript при отмене пишет в stderr `execution error: User canceled. (-128)` — вывод непустой, поэтому ветка «Отменено…» недостижима, и пользователь увидит сырой английский текст ошибки.

**BR-05. `pkill -x pppd` при отключении убивает ВСЕ pppd в системе**
- Северити: Medium. Отключение снесёт и чужие PPP-сессии (системный L2TP/PPPoE, CLI-скрипт l2tp-connect.sh), а не только туннель приложения.

**BR-06. Устаревший pidfile → kill постороннего процесса**
- Северити: Medium. `/var/run/l2tp-office-app.pid` не проверяется на принадлежность pppd. После перезагрузки PID может достаться другому процессу — connect/disconnect пошлёт ему TERM.

**BR-07. «Подключено» показывается для любого ppp0, не только своего**
- Северити: Medium. Статус: подтверждён (туннель, поднятый извне, приложение показало как свой). Если ppp0 поднят CLI-скриптом или другим PPP-клиентом, приложение отобразит «Подключено» с чужими IP; «Отключить» убьёт чужую сессию (см. BR-05).

**BR-08. Disabled-состояние синей кнопки «Подключить» почти неотличимо**
- Северити: Medium (UX). Статус: подтверждён скриншотами.
- Факт: в заблокированном состоянии фон остаётся насыщенно-синим, приглушается только текст. При беглом взгляде кнопка выглядит активной. Сравнение доступно только при виде обеих состояний рядом.

**BR-09. SIGTERM не всегда завершает pppd**
- Северити: Medium. Статус: наблюдалось живьём — pppd не завершился за ~9 с после TERM, потребовался KILL. Disconnect-скрипт ждёт 5 с и добивает KILL — сценарий покрыт, но время отключения может достигать 6-8 с без индикации причины.

### Низкие / Low

**BR-10. Keychain перезаписывается на каждый введённый символ пароля** — `didSet` на SecureField-биндинге делает delete+add в Keychain на каждый keystroke; случайное редактирование поля мгновенно затирает сохранённый пароль (тест поля из-за этого риска намеренно не проводился деструктивно).

**BR-11. Пароль во временном файле** — `/etc/ppp/l2tp-office-app.opts` (root, 600) удаляется в конце скрипта, но при аварийном прерывании (kill osascript, перезагрузка в момент подключения) файл с паролем открытым текстом останется на диске.

**BR-12. Нет валидации поля «Сервер»** — принимается любой мусор; результат — невнятный «pppd завершился с ошибкой» вместо «неверный адрес сервера». Значение с пробелом сломает opts-файл (не квотируется).

**BR-13. В режиме «Только выбранные сети» не используется DNS сервера** — `usepeerdns` добавляется только в режиме «Весь трафик»; имена хостов офисной сети не резолвятся, в UI это никак не объяснено.

**BR-14. Выход из приложения при активном туннеле** — «Выход» завершает приложение молча, туннель остаётся висеть без предупреждения; управлять им до перезапуска приложения нечем.

**BR-15. Автоподключение при входе в систему** — сразу после логина внезапно появляется системный диалог пароля без контекста, откуда он.

**BR-16. Статус-индикатор при «Отключаю…» остаётся зелёным** — точка красится по isConnected раньше, чем по busy: во время отключения зелёная точка + текст «Отключаю…» (смешанный сигнал; в панели menu bar так же).

**BR-17. Ошибка «Отменено (нет прав администратора)»** — формулировка вводит в заблуждение: пустой вывод скрипта не обязательно отмена.

### Замечания / оптимизации

- OPT-1: поллинг `ifconfig ppp0` раз в 2 с спавнит процесс постоянно (и при закрытом окне). Лучше `NWPathMonitor`/`SCDynamicStore` с колбэками, поллинг лога — только при открытом окне.
- OPT-2: `lastError` не очищается при старте новой попытки — старая ошибка видна рядом со статусом «Подключаюсь…».
- OPT-3: авто-скролл лога не срабатывает при первом открытии окна (только на изменение текста).
- OPT-4: **ИСПРАВЛЕН в v1.50** — сборка universal arm64+x86_64 (`swiftc -target ...` + `lipo`, Go arm64+amd64 + `lipo`).
- OPT-5: лог pppd в `/tmp` теряется при перезагрузке; для разборов лучше `~/Library/Logs/`.
- NOTE-1: клик по иконке приложения в menu bar недоступен автоматизации (элемент хостится процессом Control Center, гранта на который нет) — кнопки панели «Подключить/Отключить/Открыть окно/Выход» требуют ручной проверки. Иконка (открытый/закрытый замок) синхронизируется корректно.
- NOTE-2: горизонтальное масштабирование ОК: на минимальной ширине тексты не обрезаются, лог переносится, кнопки сжимаются корректно.
- NOTE-3: сверка UI↔система ОК: внешнее падение/убийство туннеля приложение отражает за ~2-4 с; поля настроек корректно блокируются на время подключения.

## Тест-сессия 2026-07-17 (v1.1 → v1.2): подключение/отключение, маршрутизация, восстановление сети

Методика: фоновый непрерывный `ping 8.8.8.8` на протяжении всей сессии (контроль целостности существующих соединений), до/после каждого шага — `route -n get`, `scutil --dns`, `curl https://…`, `nslookup`.

**BR-18 (КРИТИЧЕСКИЙ, исправлен в v1.2). После отключения пропадает интернет** — pppd при teardown не всегда откатывает закоммиченный сетевой стор (`Committed PPP store`). В `State:/Network/Service/*/IPv4` остаются мёртвые PPP-сервисы с `InterfaceName: ppp0`, configd продолжает считать один из них Primary. Симптомы: default route деградирует до interface-scoped (`route -n get default` → «not in table», флаг `I` в netstat), `State:/Network/Global/DNS` пустеет, `/var/run/resolv.conf` без nameserver — ICMP по старым склонированным маршрутам ещё ходит, всё остальное (TCP/UDP/DNS) мертво. Прежняя restore-логика в disconnect-скрипте не помогала: проверка `route -n get default` выполнялась до того, как configd переигрывал конфигурацию (гонка), и не трогала DNS вообще.
Фикс v1.2 (connect- и disconnect-скрипты): 1) зачистка стейл-ключей `State:/Network/Service/*/IPv4` с ppp0 (только когда интерфейса ppp0 реально нет); 2) восстановление глобального default route с 5 повторами (delete+add через шлюз из DHCP); 3) если `State:/Network/Global/DNS` пуст — обновление DHCP-аренды на активном en*-интерфейсе (только если он реально на DHCP); 4) flush кэша DNS. Проверено: после disconnect в v1.2 primary-сервис, маршруты, DNS и HTTPS восстанавливаются полностью, стейл-ключей не остаётся.

**BR-19 (инфраструктура, не приложение). В режиме «Весь трафик через VPN» интернет через сервер фактически не работает** — клиентская маршрутизация корректна (default через ppp0, `route get` подтверждает), но через туннель проходит только мелкий ICMP (56 байт, 0% потерь); ICMP ≥1300 байт — 100% потерь, TCP (443) — таймаут, UDP/DNS — таймаут. Похоже на отсутствие NAT/форвардинга TCP/UDP или жёсткий фаервол на стороне VPN-сервера 213.79.84.225. Приложение виновато лишь косвенно: стоит показывать предупреждение, если после подключения в режиме «Весь трафик» проверка доступности (например, TCP до 1.1.1.1:443) не проходит.

**BR-20 (исправляется в v1.13). TCP через Apple `L2TP.ppp` не устанавливается, хотя ICMP/traceroute работают** — в split-режиме `tcpdump` на `ppp0` видит валидный TCP `SYN` и ответный `SYN-ACK` от `172.16.99.5:8080`, checksums корректные, MTU не превышен, но macOS TCP-счётчики не видят входящий `SYN-ACK`, соединение остаётся в `SYN_SENT`. ECN, SACK, MPTCP, route-варианты, `looplocal`, `SC_LOOP_LOCAL`, локальный host-route и Tailscale как причина исключены. SystemConfiguration L2TP без IPsec требует глобальный `/etc/ppp/options` и всё равно не лечит TCP, поэтому v1.13 переводит основной путь на user-space L2TP helper: `pppd pty` + собственная UDP/L2TPv2 инкапсуляция.

### Новые баги

**BR-24 (закрыт: BY DESIGN, решение владельца). Пароль VPN хранится в UserDefaults в base64, а не в Keychain**
- Статус 2026-07-17: владелец проекта подтвердил, что такое хранение — осознанное решение, багом не считается. Не репортить повторно.
- Факт: `vpnPasswordLocal` в `~/Library/Preferences/com.rokot.l2tp-office.plist` — base64 от пароля; `defaults read … | base64 -d` возвращает пароль открытым текстом. Подтверждено на живой системе.
- base64 — кодирование, не шифрование. Любой процесс пользователя (или бэкап plist) читает пароль.
- Это регресс относительно заявленного фикса BR-10 («Keychain с дебаунсом 1 с»): текущий код Keychain не использует вообще, `persist()` пишет plist на каждый нажатый символ.

**BR-25 (Medium). Рассинхронизация версий root-helper: лог не очищается при запуске + внезапный админ-диалог на «Очистить лог»**
- Факт 1: после запуска приложения показан лог прошлой сессии (файл `/tmp/l2tp-office-app.log` root:wheel, mtime старый). `clearLogFileLocally()` молча падает на root-owned файле, а root-helper-очистка пропускается, т.к. `rootHelperInstalled()` вернул false.
- Факт 2: причина false — установленный helper был v1.49 при требуемой 1.50. Первое же привилегированное действие (клик «Очистить лог») вызвало переустановку helper-а с системным диалогом пароля администратора — для пользователя это выглядит как «нажал „Очистить лог“ — система вдруг требует пароль», без какого-либо объяснения в UI.
- Систематика: `build-install.command` не обновляет `/Library/PrivilegedHelperTools`, поэтому после каждого поднятия версии первый privileged-вызов будет требовать пароль. Нужно предупреждение в UI («требуется обновление системного компонента») либо явный шаг установки.

**BR-26 (Medium, UX). Пустой логин/пароль: кнопка «Подключить» заблокирована без объяснения**
- Живой тест: сервер/пароль/сети валидны, логин пуст → кнопка неактивна, никакой подписи. Для сервера и сетей красные подсказки есть, для логина/пароля — нет. Пользователь не может понять, почему нельзя подключиться.

**BR-27 (Low, UX). ⌘N плодит дубликаты главного окна**
- File → «New L2TP Office Window» (⌘N) открывает второе (третье, …) идентичное окно — подтверждено живым тестом. Для утилиты-синглтона нелогично; `showMainWindow()` из панели вернёт первое попавшееся окно. Причина: `WindowGroup` вместо `Window` (доступен с macOS 13).

**BR-28 (Low). Меню приложения не локализовано**
- «New L2TP Office Window», «Close», File/Edit/View/Window/Help — английские при полностью русском UI.

**BR-29 (Medium, код, Go-helper). UDP-reconnect в helper-е фактически неработоспособен + data race**
- `reconnectUDP()` заменяет `c.conn` и закрывает старый сокет, но `readLoop` заблокирован в `Read` на старом сокете → получает ошибку закрытого сокета → пишет в `errs` → мост завершается, туннель падает. То есть «переживание» смены сети внутри helper-а не работает — спасает только внешний реконнект приложения.
- Плюс гонки: `c.conn` заменяется без синхронизации; `nr`/`ns` пишутся из `readLoop` и читаются из `sendControl`/`sendAck` конкурентно (`sendAck` вообще без мьютекса).

**BR-30 (Low, код). Опечатка в probe-адресе сигнатуры пути: `129.0.0.1`**
- `currentSystemPathSignature()` делает `route -n get` для `default`, `1.1.1.1` и `129.0.0.1`. Последний похож на опечатку (127.0.0.1? 1.0.0.1?) — это чужой публичный диапазон. Каждые 5 с выполняется route-запрос к случайному адресу.

**BR-31 (Low/Medium, код). Автопереподключение по ICMP-ping может уйти в вечный цикл**
- 2 неудачных пинга PPP-peer подряд (интервал 5 с) → полный disconnect+connect. Если сервер перестанет отвечать на ICMP (фильтр/нагрузка) при живом туннеле — приложение будет бесконечно пересоздавать рабочее соединение. Рекомендуется более надёжный критерий (данные/TCP) или больший порог с backoff.

**BR-32 (Low, код). AUTHFAIL определяется grep-ом по всему логу**
- Лог не очищается между попытками внутри сессии: старая запись «CHAP fail» от предыдущей попытки пометит новую нейтральную ошибку как «Ошибка аутентификации».

**BR-33 (Low, код). Апдейтер: приложение завершается до подтверждения установки**
- `runInstaller` делает `NSApp.terminate` сразу после запуска osascript-установщика. Отмена админ-диалога установки → приложение уже закрыто, обновление не установлено, никакого сообщения. При активном туннеле terminate дополнительно запустит disconnect поверх установки (гонка двух privileged-операций).

**BR-34 (Low, безопасность). Файл с паролем живёт всё время соединения**
- `/etc/ppp/l2tp-office-app.opts` (пароль открытым текстом, root 0600) создаётся при connect и удаляется только при disconnect — при жёстком выключении/панике останется на диске. `trap … EXIT` из фикса BR-11 в текущем root-helper-е отсутствует. При штатном disconnect удаление подтверждено живым тестом.

**BR-35 (Info, код). `cleanup_ppp_state` не проверяет отсутствие ppp0**
- В отличие от описания фикса BR-18 («только когда интерфейса ppp0 реально нет»), текущий код удаляет State:-ключи любого ppp0 безусловно. При гонке с чужим PPP-туннелем (foreignTunnel появился между refresh и кликом, либо emergencyStop) можно снести конфигурацию чужого туннеля.

### Замечания / прочее

- OPT-1 остаётся: поллинг `ifconfig`/`pgrep` каждые 2 с + `route -n get` ×3 каждые 5 с — постоянный спавн процессов даже при закрытом окне.
- OPT-5 остаётся: лог в /tmp теряется при перезагрузке.
- OPT-6 (новое): пустой лог отображается пустой областью; подпись «Лог пока пуст.» показывается только если файла нет вообще (для пустого файла — просто пустота).
- OPT-7 (новое): подписи-ошибки под полями динамически сдвигают нижележащие элементы на ~18 px — при появлении/исчезновении подписи layout «прыгает» (и легко промахнуться кликом). Лучше резервировать место под подпись.
- Проверка обновлений: ручная и автоматическая проверка против GitHub API работает («Актуальная версия», latest = v1.50 с asset-ом). Полный цикл установки обновления не проверялся — нет релиза новее установленного.
- Не проверено живьём в этой сессии: AUTHFAIL (пароль намеренно не менялся), BR-14 (диалог выхода при активном туннеле — по коду присутствует), «Остановить подключение» (emergencyStop), тумблеры «Запускать при входе»/«Автоподключение» (чтобы не менять системную регистрацию SMAppService), установка обновления поверх релиза.

### Итог сессии

Заявленные фиксы v1.38–v1.50 в подавляющем большинстве подтверждены живыми тестами; сетевые функции (connect/split-маршрутизация/disconnect/восстановление) работают корректно, целостность стороннего трафика подтверждена (207/207 пингов). Главные новые находки: рассинхрон версий root-helper с внезапными админ-диалогами (BR-25) и отсутствие подсказки при пустом логине/пароле (BR-26). BR-24 (пароль в UserDefaults) закрыт как by design по решению владельца.
